# NFR Requirements - Unit 3: Order Domain

## 개요
Unit 3 (주문 도메인)의 비기능 요구사항을 정의한다. requirements.md의 NFR-01~05를 기반으로 Unit 3 특화 요구사항을 상세화한다.

---

## 1. 성능 요구사항 (Performance)

### NFR-U3-P01: 주문 생성 API 응답 시간
- **목표**: 1초 이내 (P95)
- **측정 범위**: 클라이언트 요청 수신 ~ 응답 전송 완료
- **포함 작업**: 입력 검증, 세션 조회, 메뉴 가격 검증, 주문번호 생성, 트랜잭션(Order+OrderItem 생성), SSE 브로드캐스트
- **근거**: requirements.md NFR-01 기준

### NFR-U3-P02: 주문 상태 변경 API 응답 시간
- **목표**: 500ms 이내 (P95)
- **포함 작업**: 주문 조회, 상태 업데이트, SSE 브로드캐스트 (관리자 + 테이블)

### NFR-U3-P03: 주문 조회 API 응답 시간
- **목표**: 500ms 이내 (P95)
- **포함 작업**: 세션/날짜 기반 주문 조회 + OrderItem JOIN

### NFR-U3-P04: SSE 이벤트 전달 지연
- **목표**: 2초 이내 (이벤트 발생 ~ 클라이언트 수신)
- **근거**: requirements.md NFR-01 (SSE 실시간 주문 알림 2초 이내)

### NFR-U3-P05: 이용 완료 처리 응답 시간
- **목표**: 2초 이내 (P95)
- **포함 작업**: 세션 주문 조회, OrderHistory 복사, 원본 삭제, 세션 종료, SSE 브로드캐스트
- **참고**: 주문 수에 비례하여 시간 증가 가능 (테이블당 최대 ~50건 주문 가정)

---

## 2. 가용성 요구사항 (Availability)

### NFR-U3-A01: SSE 연결 안정성
- **목표**: 매장 영업 시간(11:30~22:00) 동안 SSE 연결 안정 유지
- **Heartbeat**: 30초 간격 heartbeat 전송
- **연결 실패 처리**: heartbeat 전송 실패 시 즉시 연결 제거 (리소스 누수 방지)
- **재연결**: EventSource 기본 자동 재연결 동작 활용

### NFR-U3-A02: 서버 재시작 복원
- **목표**: 서버 재시작 후 SSE 클라이언트 자동 재연결
- **세션 유지**: JWT 기반 인증으로 서버 재시작 시에도 세션 유지
- **SSE 재연결**: 클라이언트 EventSource가 자동 재연결, 필요 시 API로 최신 데이터 조회

---

## 3. 확장성 요구사항 (Scalability)

### NFR-U3-SC01: 동시 접속 처리
- **목표**: 테이블 50개 + 관리자 5명 동시 SSE 연결 처리
- **SSE 연결 수**: 최대 55개 동시 연결 (테이블 50 + 관리자 5)
- **관리 방식**: 단순 인메모리 Map (MVP 수준)
- **근거**: 중규모 매장 기준, 단일 서버로 충분

### NFR-U3-SC02: 데이터 증가 대응
- **주문 데이터**: 일일 최대 ~500건 주문 (50테이블 × 10회전 × 1건/회전 가정)
- **OrderHistory**: 90일 보존, 조회 시 WHERE 필터링
- **인덱스**: store_id + session_id, store_id + ordered_at 복합 인덱스로 조회 성능 확보

### NFR-U3-SC03: Store ID 기반 멀티테넌트
- **현재**: 단일 매장
- **확장 준비**: 모든 쿼리에 store_id 조건 포함
- **근거**: requirements.md NFR-03

---

## 4. 신뢰성 요구사항 (Reliability)

### NFR-U3-R01: 주문 생성 트랜잭션
- **목표**: Order + OrderItem 생성의 원자성 보장
- **방식**: SQLite 단일 트랜잭션
- **실패 시**: 자동 롤백, 서버 측 재시도 최대 2회 후 에러 반환

### NFR-U3-R02: 이용 완료 트랜잭션
- **목표**: OrderHistory 복사 + 원본 삭제 + 세션 종료의 원자성 보장
- **방식**: SQLite 단일 트랜잭션
- **실패 시**: 자동 롤백, 서버 측 재시도 최대 2회 후 에러 반환

### NFR-U3-R03: 주문 삭제 트랜잭션
- **목표**: Order + OrderItem 삭제의 원자성 보장
- **방식**: CASCADE 삭제 또는 트랜잭션 내 순차 삭제

### NFR-U3-R04: SSE 이벤트 전달 보장
- **수준**: Best-effort (최선 노력)
- **미전달 시**: 클라이언트가 API로 최신 데이터 조회
- **근거**: MVP 수준, Last-Event-ID 미구현

---

## 5. 유지보수성 요구사항 (Maintainability)

### NFR-U3-M01: 로깅
- **수준**: 최소 로깅 (에러만 기록)
- **방식**: console.error 사용
- **대상**: 트랜잭션 실패, 예외 발생, SSE 전송 실패
- **형식**: 타임스탬프 + 에러 메시지 + 스택 트레이스

### NFR-U3-M02: 에러 처리
- **방식**: 글로벌 에러 핸들러 (Unit 1의 errorHandler 활용)
- **응답 형식**: `{ error: { code: string, message: string } }`
- **HTTP 상태 코드**: 적절한 4xx/5xx 코드 사용

### NFR-U3-M03: 테스트
- **프레임워크**: Vitest
- **범위**: Service 레이어 단위 테스트, Repository 레이어 단위 테스트
- **목표**: 핵심 비즈니스 로직 커버리지

### NFR-U3-M04: 코드 구조
- **패턴**: Controller → Service → Repository 레이어드 아키텍처
- **원칙**: 단일 책임, 의존성 주입 (생성자 주입)
- **타입 안전성**: TypeScript strict 모드

---

## 6. 데이터 무결성 요구사항 (Data Integrity)

### NFR-U3-D01: 가격 무결성
- **규칙**: 서버에서 DB 가격 조회 후 저장 (클라이언트 가격 무시)
- **스냅샷**: 주문 시점의 메뉴명, 단가를 OrderItem에 저장

### NFR-U3-D02: 주문번호 고유성
- **규칙**: store_id + 날짜 기준 순번 고유
- **보장**: SQLite 단일 쓰기 특성으로 동시성 충돌 방지

### NFR-U3-D03: 세션 무결성
- **규칙**: 테이블당 활성 세션 최대 1개
- **보장**: 세션 생성 전 기존 활성 세션 확인
